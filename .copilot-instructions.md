# UseClevr AI Assistant Instructions

You are assisting with UseClevr, an AI-powered business intelligence application.

## Project Context
- **Framework**: Next.js 16 (app router), React 19, TypeScript 6, Tailwind CSS
- **Database**: Neon PostgreSQL with Drizzle ORM
- **AI**: Gemini via AI SDK (cloud) + local AI routes
- **Package Manager**: pnpm
- **Verification**: `pnpm build` is the preferred smoke test

## Key Files to Review First
- `AGENTS.md` - Project-specific guidance for AI agents
- `ai-chat-behavior.config.ts` - Shared behavior presets and coding conventions
- `.aiignore` - Files/folders to ignore to save tokens

## Rules for Changes
1. **Keep changes focused** on the requested behavior only
2. **Respect `.aiignore`** — don't load: `.next/`, `dist/`, `.git/`, `node_modules/`, pnpm-lock.yaml
3. **Don't edit generated files** in `.next/` or `dist/` unless explicitly asked
4. **Verify with**: `pnpm build` (not `pnpm exec tsc --noEmit`)
5. **Use existing patterns** — prefer components in `src/components/` before creating new ones
6. **Keep provider secrets server-side only** — use same-origin API routes for AI work
7. **Use Tailwind tokens** and respect light/dark theme contrast

## Project Commands
```bash
pnpm dev       # Development server
pnpm build     # Production build (smoke test)
pnpm start     # Production server
pnpm db:push   # Migrate database schema
pnpm db:studio # Open Drizzle Studio
```

## AI Behavior
- Deterministic TypeScript/query logic is the source of truth for metrics
- AI must explain verified data, never invent metrics
- Log failures enough to debug without leaking prompts or data
- If required data is missing, state what's missing and offer next steps

## File Structure
- **Routes**: `src/app/`
- **Components**: `src/components/` (reusable) + `src/components/ui/` (base UI)
- **Business Logic**: `src/lib/`
- **Static Assets**: `src/assets/`
- **Styles**: `src/assets/styles/globals.css` + Tailwind

## When You See These Patterns
- ✅ Use `const` and narrow helpers
- ✅ Keep server work in route handlers/actions
- ✅ Keep client components focused on UI state
- ✅ Handle failed requests with user-facing errors
- ❌ Don't silently swallow errors
- ❌ Avoid ad hoc string parsing (use schema validation instead)
